# Day 01 — JVM & Memory

## Question 1

### Explain the JVM architecture.

### Answer

The JVM is responsible for running Java applications. When we write Java code, it's first compiled by `javac` into bytecode, which is platform independent.

When the application starts, the JVM loads the required classes using the Class Loader. It then creates the runtime memory areas like the Heap, Stack, and Metaspace.

The Execution Engine starts executing the bytecode. Initially, it interprets the code, but if some methods are executed frequently, the JIT compiler compiles them into native machine code to improve performance.

The JVM also manages memory automatically using the Garbage Collector, so developers don't need to free memory manually.

---

## Question 2

### Why is Java platform independent?

### Answer

Java is platform independent because the Java compiler generates bytecode instead of native machine code.

The bytecode can run on any operating system that has a compatible JVM. So we compile the application once, and the JVM translates the bytecode into native instructions for the underlying operating system.

That's why we often say "Write Once, Run Anywhere."

---

## Question 3

### What is bytecode?

### Answer

Bytecode is the intermediate code generated by the Java compiler after compiling a `.java` file.

It's not specific to any operating system or processor. The JVM understands this bytecode and executes it by interpreting it or compiling it into native machine code using the JIT compiler.

This is one of the main reasons Java applications are portable across different platforms.

---

## Question 4

### What is the difference between the Interpreter and the JIT compiler?

### Answer

The Interpreter executes bytecode instruction by instruction, so it can start running the application immediately.

The JIT compiler monitors the application while it's running. If it detects that a method is executed many times, it compiles that method into native machine code.

This means the first execution may be slower, but repeated executions become much faster because the JVM no longer needs to interpret the bytecode.

The Interpreter provides fast startup, while the JIT compiler improves long-term performance.

---

## Question 5

### What is a hot method?

### Answer

A hot method is simply a method that is executed frequently.

The JVM keeps track of how often methods are called. When a method becomes "hot," the JIT compiler compiles it into native machine code.

This optimization improves performance because native code executes much faster than interpreted bytecode.

---

## Question 6

### What is the difference between Stack and Heap?

### Answer

The easiest way to think about it is that every thread has its own Stack, while the Heap is shared by the entire application.

The Stack stores method calls, local variables, primitive values, and references to objects. Every time a method is called, a new Stack Frame is created.

The Heap stores the actual objects and arrays. Objects stay there until they are no longer referenced and the Garbage Collector removes them.

Because the Stack is very simple and follows a LIFO structure, it's much faster than the Heap.

---

## Question 7

### Where are local variables stored?

### Answer

Local variables are stored inside the current method's Stack Frame, which is located on the thread's Stack.

Each thread has its own Stack, so local variables are naturally isolated from other threads.

For object variables, the reference is stored on the Stack, while the actual object is stored on the Heap.

---

## Question 8

### Where are object references stored?

### Answer

Object references are usually stored inside the Stack Frame as local variables.

The reference simply points to the object's location in the Heap.

So the reference is on the Stack, while the actual object lives on the Heap.

---

## Question 9

### Where are objects stored?

### Answer

Objects and arrays are stored on the Heap.

The Heap is shared by all threads, which allows different threads to access the same objects when needed.

Objects remain on the Heap until they become unreachable, at which point the Garbage Collector can reclaim the memory.

---

## Question 10

### What is Metaspace?

### Answer

Metaspace is the JVM memory area that stores class metadata, such as class definitions, method information, field metadata, annotations, and the runtime constant pool.

It replaced PermGen in Java 8 and uses native memory instead of Heap memory.

One important thing to remember is that Metaspace stores information about classes, not the actual objects created from those classes.
